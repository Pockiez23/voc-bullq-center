# 🤖 System Architecture & Agents Workflow (AGENTS.md)

เอกสารนี้อธิบายสถาปัตยกรรม บทบาทหน้าที่ และกลยุทธ์การตั้งเวลา (Timing Strategy) ของเอเจนต์ที่ทำงานอยู่เบื้องหลังในโปรเจกต์ `voc-bullq-center` ซึ่งเป็นระบบจัดการคิวงานแบบอะซิงโครนัส (Asynchronous Job Queue System) สำหรับรองรับการส่งข้อมูลไปยังระบบภายนอกผ่านโปรโตคอล SOAP

---

## 📌 ภาพรวมสถาปัตยกรรมระบบ (System Overview)

ระบบถูกออกแบบมาโดยใช้หลักการ **Decoupling** แยกส่วน "ผู้สั่งงาน" (Center) และ "ผู้ทำงาน" (Worker) ออกจากกัน เพื่อป้องกันไม่ให้คอขวด (Bottleneck) ไปกระทบกับระบบหลัก:

* **Source of Truth:** `PostgreSQL` (ผ่าน Prisma) ทำหน้าที่เก็บข้อมูลและสถานะของงานที่แท้จริง
* **Message Broker:** `Redis` (ผ่าน BullMQ) ทำหน้าที่เป็นคิวตรงกลาง ช่วยพักข้อมูลและจัดการลำดับงาน

---

## 1️⃣ Center Agent (Producer / Manager)

Center ทำหน้าที่เป็น **"ผู้จัดการคิว"** ที่ไม่ลงมือทำงานหนักเอง แต่จะคอยสอดส่อง Database เพื่อจ่ายงานให้ Worker และคอยตรวจสอบความผิดปกติของระบบ

* **📍 ตำแหน่งไฟล์:** `src/cron/job.cron.ts`
* **▶️ คำสั่งรัน:** `bun run dev` หรือ `bun run center`

### ⚙️ กลไกการทำงานและการตั้งเวลา (Cron Strategies)

| งานหลัก (Tasks) | รอบการทำงาน (Interval) | หน้าที่และเหตุผล |
| :--- | :--- | :--- |
| **Job Dispatcher**<br>(ระบบจ่ายงานใหม่) | **ทุกๆ 1 นาที** | **หน้าที่:** มองหางานสถานะ `PENDING` เช็คว่าปัจจุบันมี Worker ว่างอยู่หรือไม่ (นับ `RUNNING` == 0) ถ้าว่างจะหยิบงานมา 1 ชิ้น โยนลง Redis และเปลี่ยนสถานะ DB เป็น `RUNNING` เพื่อล็อกงาน<br><br>**เหตุผล (Why 1 Min?):** เพื่อลด "เวลาสูญเปล่า (Idle Time)" เมื่อ Worker ทำงานเสร็จ จะมีช่วงรอคอยงานใหม่สูงสุดไม่เกิน 1 นาที ทำให้ระบบไหลลื่นและได้ Throughput ที่ดีที่สุด |
| **Rescue Stuck Jobs**<br>(ระบบกู้ชีพงานค้าง) | **ทุกๆ 15 นาที** | **หน้าที่:** ค้นหางานที่ค้างอยู่ในสถานะ `RUNNING` นานเกินกว่าเวลาที่กำหนด หากพบจะทำการ Re-queue (โยนกลับเข้า Redis) และรีเซ็ตเวลาใหม่<br><br>**เหตุผล (Why 15 Mins?):** กฎของระบบกู้ชีพคือ *"เวลาตรวจสอบต้องนานกว่าเวลาทำงานจริงเสมอ"* เนื่องจาก Worker ใช้เวลาทำงาน 5 นาที การตั้งระบบกู้ชีพไว้ที่ 15 นาที (เผื่อเวลา 3 เท่า) จะช่วยป้องกันปัญหา False Alarm (ระบบเข้าใจผิดว่า Worker ตายทั้งที่แค่กำลังทำงานอยู่) |

## 2️⃣ Worker Agent (Consumer / Executor)

Worker ทำหน้าที่เป็น **"ผู้ปฏิบัติงาน"** ที่รับคำสั่งจาก Redis Queue นำไปประมวลผลให้เสร็จสิ้น และอัปเดตสถานะกลับไปยังฐานข้อมูล

* **📍 ตำแหน่งไฟล์:** `src/worker.ts`
* **▶️ คำสั่งรัน:** `bun run worker`

### ⚙️ กลไกการทำงานและวงจรชีวิต

1.  **🛡️ Pre-flight Validation (การตรวจสอบก่อนเริ่มงาน)**
    * **Ghost Job Protection:** นำ ID งานที่ได้จาก Redis ไปเช็คกับ Database เสมอ หากไม่พบข้อมูล (เช่น ฐานข้อมูลถูกรีเซ็ตแต่ไม่ได้ล้าง Redis) จะข้ามงานนั้นทันทีเพื่อป้องกัน Error
    * **Duplicate Protection:** ป้องกันการทำซ้ำโดยตรวจสอบสถานะล่าสุด หากงานเป็น `COMPLETED` หรือ `FAILED` ไปแล้ว จะเพิกเฉยต่อคิวซ้ำซ้อนนั้น
2.  **⏳ Execution (การประมวลผล)**
    * **ระยะเวลาทำงาน:** ประมวลผลและหน่วงเวลา **5 นาที ต่อ 1 งาน** (`concurrency: 1`) เพื่อป้องกันการส่ง Request ถี่เกินไปจนปลายทางรับไม่ไหว (Rate Limiting)
    * **Data Transformation:** ดึงข้อมูล Payload (JSON) มาประกอบร่างให้อยู่ในโครงสร้างมาตรฐานของ **SOAP Envelope (XML)** ตาม Requirement ของระบบปลายทาง
3.  **🚀 External Communication (การส่งมอบข้อมูล)**
    * ยิง HTTP POST ไปยัง Webhook/SOAP Service
    * ตั้งค่า Header เป็น `Content-Type: text/xml; charset=utf-8` ตามมาตรฐานของ SOAP 1.1
4.  **💾 State Management & Graceful Shutdown**
    * **อัปเดตผลลัพธ์:** หาก Webhook ตอบรับ `200 OK` จะบันทึกสถานะลง DB เป็น `COMPLETED` หากล้มเหลวจะบันทึกเป็น `FAILED`
    * **ความปลอดภัยตอนปิดระบบ:** หาก Worker ถูกสั่งปิด (`Ctrl+C` / `SIGINT`) ระหว่างทำงาน ระบบจะดักจับ Signal และไปกระชากสถานะงานที่ถืออยู่ในมือ ให้กลับมาเป็น `PENDING` คืน เพื่อให้เปิดระบบครั้งหน้าสามารถนำงานไปทำต่อได้ทันทีโดยไม่สูญหาย

## 🔄 วงจรชีวิตของงาน (Job Lifecycle State Machine)

การไหลของข้อมูลใน Database จะถูกควบคุมด้วย 4 สถานะหลัก:

* 🔵 **`PENDING`** : งานถูกสร้างขึ้นใน DB และรอให้ Center (Dispatcher) ตื่นขึ้นมาพบ
* 🟡 **`RUNNING`** : Center ล็อกงานนี้ไว้ และโยนเข้าคิวให้ Worker ทำการแปลง XML และส่ง SOAP (อยู่ในช่วงเวลา 5 นาที)
* 🟢 **`COMPLETED`** : Worker ส่ง SOAP สำเร็จ ปลายทางตอบรับเรียบร้อย จบวงจรการทำงานของงานชิ้นนั้น
* 🔴 **`FAILED`** : เกิดข้อผิดพลาดร้ายแรงที่ไม่สามารถแก้ไขได้ เช่น ปลายทางติดต่อไม่ได้ หรือ Error ฝั่งโค้ด

## 🛠 เทคโนโลยีที่ใช้ (Tech Stack)

* **Runtime:** Bun
* **Framework:** Elysia
* **Database:** PostgreSQL (ORM: Prisma)
* **Message Broker / Queue:** Redis (Library: BullMQ)
* **Protocol:** HTTP/REST (Internal) / SOAP XML (External Integration)
